git-flow的因地制宜

写在前面

  • 规范 - 当在团队开发中使用版本控制系统时，商定一个统一的工作流程是至关重要的。例如前端代码中的Eslint一样.
  • 因地制宜 - 君子生非异也,善假于物也. 需要结合实际开发的场景,制定一套双赢的规定.并且在后期实践中逐渐更新.

什么是 git-flow？

当你安装git-flow之后, 你将会拥有一些新的扩展命令。其实指令里面是预定的一些常规git操作,并且按照顺序执行。

git-flow 并不是要替代Git，它仅仅是非常聪明有效地把标准的Git命令用脚本组合了起来。

严格来讲，你并不需要安装什么特别的东西就可以使用git-flow 工作流程。你只需要了解，哪些工作流程是由哪些单独的任务所组成的，并且附带上正确的参数，以及在一个正确的顺序下简单执行那些对应的Git 命令就可以了。当然，如果你使用 git-flow 脚本就会更加方便了，你就不需要把这些命令和顺序都记在脑子里。

下面我们会逐一把各个指令进行拆解。

安装 git-flow

先在项目中(本地)建一条master分支之外的长久存在分支。建议以develop 进行命名。

在项目中执行该脚本，并且按照默认的分支名约定进行安装。尽管如此，这个安装助手还是允许你使用自己喜欢的名字。我强烈建议你使用默认的命名机制，并且一步一步地确定下去。

分支因地制宜

###master
比较常规，和以前的用法一致。

develop

长久存在的分支。顾名思义，这是一条和开发相关的分支,是你进行开发任何新的开发的基础分支。另外，该分支也汇集所有已经完成的功能，并等待被整合到 master 分支中。(这个节点比较尴尬,下面会提及该注意的地方)

feature/**

该分支在develop分支中执行git flow feature start xx 创建。
当项目开发完成的时候，这里不建议直接使用git flow feature finish xx,finish 指令会直接将开发分支xx 的合并到develop中。然后发布计划是多变的，如果执行合并指令的话，相当于develop已经被污染了，意味着不能进行新分支的开发。所以，这边我建议完成之后在dev分支merge xx。然后提测。
使用finish指令应该在测试通过之后，并且等待产品验收之前。

release

该分支恰巧连接了上述的finish场景，这时候的develop 已经是相对比较成熟的，这意味着：第一，它包括所有新的功能和必要的修复；第二，它已经被彻底的测试过了。
请注意，release分支是使用版本号命名的。这是一个明智的选择，这个命名方案还有一个很好的附带功能，那就是当我们完成了release后，git-flow 会适当地自动_去标记那些release 提交。

最后是完成release

这个操作比较危险，具体执行的操作如下:

  • 首先，git-flow 会拉取远程仓库，以确保目前是最新的版本。
  • 然后，release的内容会被合并到 master 和 develop 两个分支中去，这样不仅产品代码为最新的版本，而且新的功能分支也将基于最新代码。
  • 为便于识别和做历史参考，release提交会被标记上这个release的名字（在我们的例子里是 1.1.5(tag)）。
  • 清理操作，版本分支会被删除，并且回到 develop。

    hotfix

    当线上出现bug的时候,需要从master上拉取出最新的代码，并且进行fix

这个指令所创建的代码是基于master的。和release分支区别的地方就在于来源分支不一样。
当完成bug fix的时候。

  • 完成的改动会被合并到 master 中，同样也会合并到 develop 分支中，这样就可以确保这个错误不会再次出现在下一个release 中。
  • 这个hotfix 程序将被标记起来以便于参考。
  • 这个hotfix 分支将被删除，然后切换到 develop 分支上去。

    最后

    上述流程中集合了自动合并到master的过程，但是在开发环境中的dev分支和docker分支需要自行合并，并手动或者自己执行发布脚本。
